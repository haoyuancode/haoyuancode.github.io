JWT

介绍

Jwt全称是JSON Web Tokens

使用json的形式作为web应用中的令牌，将需要传输的信息(比如用户名和密码)作为json对象进行传输，在传输过程中还可以进行加密、认证授权操作。

应用场景

  1. 授权

    最常用的场景。用户登录过后，以后的客户端请求中都会包含JWT数据，一方面不用每次都输入账号和密码，另一方面允许用户访问令牌中允许的路由、服务和资源。常用的单点登录就是使用JWT的一项功能。可以实现跨域。

  2. 交换信息

    JWT可以进行签名，发送客户端和服务器之间发送一次请求后，会使用算法产生一个签名。即使遭到篡改，但签名是通过算法进行记录的，更改后会有很大的改变，服务器和客户端都会验证第一次生成的签名，所以使用签名可以验证是否遭到篡改。

JWT和session的区别

对jwt、session认证之间的区别还是不太清楚，都是个人的理解

用户登录成功后，服务器会生成一个sessionId返回给客户端，session保存到服务器的内存中。session的过期时间等其他可以自由设置。
客户端再次发送请求就会携带sessionId去和服务器内存中的sessionid进行比较，如果存在则通过。
如果是集群，存在服务器中的sessionid要进行集群中的共享，加大了内存的占用。

JWT是用户登录成功，服务器会生成一个token，将token返回给客户端并保存到客户端中。客户端之后再次访问会将JWT放在请求头中，同时解决了伪造跨域攻击，后端验证token是否存在、过期等，并返回相应的结果。

JWT结构

是一个字符串，由三部分组成：

组成样式例如: xxx.xxx.xxx，三部分之间由.隔开

  1. 标头(Header)

    由令牌的类型(typ：)和使用的签名算法(alg：)组成。

    {

    “alg”: “HS256”,

    ​ “typ”:”JWT”

    }

    上边JSON形式的数据会经过Base64进行编码，变成一段字符串，生成的字符串成为结构中的第一部分标头。
    经过Base64编码之后的字符串还可以使用Base64解码变成最初的样子。

  2. 有效载荷(PayLoad)

    简单来说就是用来存放用户登录之后的信息，比如指定用户名、密码、权限等。

    {

    ​ “name”:”mason”,

    ​ “admin”:true

    }

    同样这部分数据也是使用Base64进行编码，同样可以解码，所以不要把敏感的信息放进去，比如密码

  3. 签名(Signature)

    这部分是由标头和载荷以及自定义签名组成的。自定义的签名很重要，不要泄露。
    自定义的签名是根据标头中设置的加密算法进行加密处理生成的。

整合spring boot

整体思路

  1. 数据库中新建一张用户表，存放用户名和密码
  2. 编写entity、mapper、service等
  3. 在接口中接收传递过来的用户名和密码，并从数据库中查询验证是否存在
  4. 和数据库中的数据匹配通过，则调用jwt工具类，将用户的信息传递进去(比如用户名)，最后会生成一个token并返回
  5. 前端接受到token进行保存，以后每次访问后端时都会将token放在请求头中
  6. 后端接受请求，并从请求头中获取token进行解码比较，成功则相应请求，否则反之。
